# Renarr V1 规格说明

## 产品范围

Renarr 是一个自动化媒体库整理工具。V1 只管理 WebDAV 存储中的电影和电视剧文件，负责识别、重命名、移动、整理目录结构、生成基础 `.nfo` 和海报元数据。

V1 不播放、解码、转码、截图、下载媒体文件、下载字幕，也不管理 BT/NZB/网盘下载流程。

V1 目标命名兼容 Jellyfin/Kodi，并通过标准电影/剧集命名兼容 Infuse。

## 已确定技术栈

- TypeScript
- Tailwind CSS
- shadcn-svelte
- Zod
- SQLite
- Drizzle ORM
- Svelte Query
- Paraglide JS i18n
- Node 24
- pnpm

- `src/lib/server/*` 和 `src/worker/*` 只能在服务端/worker 使用。
- SvelteKit routes 和 worker 都不直接写复杂业务 SQL，而是调用 repository/service。

## 部署与运行时

官方部署目标是 Docker。

- SvelteKit 使用 `@sveltejs/adapter-node`。
- web 和 worker 是两个独立 Node 进程。
- 生产环境不由 web 进程 fork 或托管 worker 生命周期。
- 本地开发可以用脚本同时启动 web 和 worker。
- worker 生产环境用 `tsup` 单独构建为 JS，开发环境可用 `tsx watch`。

建议脚本：

```text
pnpm dev
pnpm dev:web
pnpm dev:worker
pnpm build
pnpm build:worker
pnpm worker
pnpm check
```

建议 Docker 形态：

```text
同一个镜像
- web command: node build
- worker command: node build-worker/index.js
```

## 核心概念

### WebDAV 来源

WebDAV 来源包含：

- kind: V1 固定为 `webdav`
- name
- url
- username
- encrypted credential

`source.name` 会用于前端展示路径 `/{source.name}`，因此不允许路径非法字符。

禁止：

```text
/ \ : * ? " < > | 控制字符
空字符串
全空白
.
..
```

### Library Path

Library path 是 WebDAV 来源内的一个管理根路径。

- 一个 source 可以有多个 library path。
- 每个 library path 固定为 `movie` 或 `tv`。
- V1 不支持混合媒体库。
- media type 创建后不可修改。
- 删除 library path 只删除本地管理记录，不删除远端文件。

### Library Item

V1 的持久化主对象是 `library_item`，不是文件级 media record。

`library_item` 代表 library root 下的一级条目：

- 一级文件夹
- 一级视频文件

身份：

```text
libraryPathId + topLevelPath
```

示例：

```text
library path: /Movies

/Movies/Inception
topLevelPath = Inception
kind = folder

/Movies/Inception.2010.mkv
topLevelPath = Inception.2010.mkv
kind = file
```

TV 示例：

```text
/TV/甄嬛传 (2011)/40.mp4

library_item 只有：
topLevelPath = 甄嬛传 (2011)
kind = folder
```

不为 `Season 01` 等更深层目录创建 library item。

### 文件级信息

V1 不保存长期 `media_files` 当前状态表。

文件级信息只存在于：

- 实时 WebDAV 读取结果
- scan 统计
- plan draft
- rename plan
- execution records
- logs

执行历史必须持久化记录文件级 source/target/status/error。

## 路径规范

内部路径统一使用 POSIX 风格。

```text
sourcePath: absolute POSIX path, starts with /
libraryPath.path: absolute POSIX path, starts with /
libraryItem.topLevelPath: relative first segment, no leading /
file path inside library: relative POSIX path, no leading /
targetRelativePath: relative POSIX path, no leading /
```

内部路径不包含 `/{source.name}`。

前端展示路径：

```text
/{source.name}/{libraryPath.path}/{libraryItem.topLevelPath}
```

`/{source.name}` 只用于展示，不传给 FileClient，不参与 DB identity。

## FileClient 抽象

业务层使用存储抽象 `FileClient`，V1 只有 WebDAV 实现。

```ts
interface FileClient {
  listDirectory(path: string): Promise<FileEntry[]>;
  ensureDirectory(path: string): Promise<void>;
  moveFile(from: string, to: string, options: MoveOptions): Promise<MoveResult>;
  deletePath(path: string): Promise<void>;
  exists(path: string): Promise<boolean>;
}
```

实现：

```text
WebDavFileClient implements FileClient
```

V1 UI 仍然称为 WebDAV 来源，不提前泛化成其它存储。

底层 WebDAV 推荐使用 `webdav` npm 包，并通过自有 adapter 隔离。

## 凭据与安全

V1 无登录系统、无用户系统。如果暴露到公网，应由反向代理、Authelia 等部署工具处理认证。

WebDAV 密码/token 本地加密保存。

- 使用 AES-256-GCM。
- `RENERR_SECRET_KEY` 是 32 字节 base64 key。
- 每次加密使用随机 12-byte IV。
- 前端永不返回或展示 plaintext credential。

编辑 WebDAV source：

- password/token 输入框默认空。
- 不填表示保留旧密文。
- 填写新值则重新加密覆盖。
- 连接测试如果表单有新密码则用新密码，否则解密已保存 credential。

## WebDAV 浏览与连接测试

Library setup 的 WebDAV 浏览：

- directory-by-directory
- 只显示目录
- 不递归加载整棵树
- 当前目录可保存为 movie/tv library path

Item detail：

- 实时读取 WebDAV
- 展示视频文件、sidecar、metadata 文件等

连接测试：

- 只测试认证和 list/propfind。
- 不测试写权限。
- 不创建、移动、删除任何远端文件。

排序：

- 目录优先
- 自然排序
- 前端使用 `Intl.Collator('zh-CN', { numeric: true, sensitivity: 'base' })`

## 扫描规则

### 支持的视频扩展名

```text
.mkv .mp4 .avi .mov .m4v .ts .m2ts .wmv
```

V1 不支持：

- archives
- ISO
- BDMV folder
- DVD folder structure

普通 `.m2ts` 文件在允许深度内可作为视频文件。

### 扫描深度

Movie library：

```text
libraryRoot/file.ext
libraryRoot/folder/file.ext
```

TV library：

```text
libraryRoot/file.ext
libraryRoot/folder/file.ext
libraryRoot/folder1/folder2/file.ext
```

### scan_library_path

`scan_library_path` 以远端 library root 一级目录为事实来源，同步 `library_items` 当前表。

流程：

1. list library root 一级条目。
2. 过滤一级文件夹和一级支持的视频文件。
3. 新 topLevelPath 创建 library item。
4. 仍存在的 topLevelPath 保留 identity/status 并按状态处理。
5. 远端不存在的 topLevelPath 硬删除本地 item。

状态处理：

```text
unidentified
- 每次扫描尝试识别
- high confidence -> identified 或后续自动整理
- fuzzy/ambiguous/no_match/parse_failed -> pending_review

pending_review
- 扫描跳过
- 不刷新候选
- 不检查内部文件
- 等用户手动指定或重新识别

identified
- 扫描不再查 TMDB
- 不执行 scan_library_item
- 需要先整理

organized
- 跳过 TMDB 查询
- 检查内部文件统计
- 如果 auto organize enabled 且存在可明确处理的 non-compliant 文件，可生成自动 rename plan

failed
- 扫描跳过
- 仅 root 同步时如果远端路径消失则硬删除
- 用户必须手动处理
```

### scan_library_item

任务名为：

```text
scan_library_item
```

它表示手动扫描单个 `library_item`。

只支持状态：

- `organized`
- `unidentified`

不支持：

- `pending_review`: 必须先手动指定 identity。
- `failed`: 必须先处理失败。
- `identified`: 必须先整理。

`scan_library_item` 不创建二级 library item。

## Library Item 状态

状态：

```text
unidentified
pending_review
identified
organized
failed
```

语义：

```text
unidentified
- 当前一级条目尚未识别

pending_review
- 扫描已经处理过，但需要用户介入
- 可能有候选，也可能没有候选
- reviewReason 可为 fuzzy、ambiguous、no_match、missing_episode、parse_failed

identified
- 已确认 TMDB identity
- 尚未整理

organized
- 顶级文件夹已匹配 identity
- 系统认为它是已管理目标目录
- 不要求内部所有文件符合当前命名模板

failed
- 最近一次关键处理失败，需要用户介入
```

模板符合度使用独立统计表达：

```text
videoFileCount
compliantFileCount
nonCompliantFileCount
unknownFileCount
```

扫描发现 organized folder 内有不符合模板的文件：

- `status` 保持 `organized`
- `nonCompliantFileCount > 0`
- UI 显示“有待处理文件”

## 识别与匹配

V1 只实现 TMDB，但数据模型保留通用字段：

```text
source
sourceMediaId
sourceMediaType
```

TMDB 使用 REST API 自封装，不使用 SDK。

建议接口：

```ts
interface TmdbClient {
  searchMovie(query, options): Promise<TmdbSearchResult[]>;
  getMovieDetails(id, options): Promise<TmdbMovieDetails>;
  searchTv(query, options): Promise<TmdbSearchResult[]>;
  getTvDetails(id, options): Promise<TmdbTvDetails>;
  getImageConfiguration(): Promise<TmdbImageConfig>;
}
```

HTTP 层提供：

- proxy env 支持
- timeout
- 小规模 transient retry
- sanitized error

不对 403/429 做复杂自动 retry。

### 媒体文件名解析

引入 `@ctrl/video-filename-parser` 作为底层解析辅助。

它只作为“文件名信号提取器”，不直接决定 high confidence。

Renarr 自己实现 wrapper：

```text
parseMoviePath()
parseTvPath()
normalizeTitle()
extractYear()
parseEpisodeFromPath()
```

高置信规则、TV 路径语义、纯数字集数等由 Renarr 自己控制。

### 高置信规则

High confidence 不等于原字符串相等，而是 normalized exact equality 加关键字段一致。

Movie high confidence：

- library type 是 movie。
- 从 folder name 或 filename 清洗出的标题，与 TMDB `title`、`original_title` 或 alternative titles 归一化后精确匹配。
- 如果文件名或目录名有年份，年份必须匹配。
- TMDB 搜索结果排名第一。
- 不存在同名同年歧义。

TV folder high confidence：

- library type 是 tv。
- 一级 folder name 与 TMDB `name`、`original_name` 或 alternative titles 归一化后精确匹配。
- 如果 folder name 有年份，first air year 必须匹配。
- TMDB 搜索结果排名第一。
- 不存在同名同年歧义。

TV root file high confidence：

- 允许参与自动高置信整理。
- 必须能解析出 show title 和 episode。
- season 可明确解析；或未解析时默认 season 1。
- TMDB TV 搜索第一结果匹配。
- TMDB detail 确认 season 存在且 `episode_count` 覆盖 episode。
- 无同名同年歧义。

示例：

```text
甄嬛传.40.mp4 -> show=甄嬛传, season=1, episode=40
```

如果 TV root file 没有 episode number，不得自动整理。

### Normalization

归一化包括：

- trim
- 全角/半角归一化
- case folding
- 标点/空白清理
- 可选简繁中文归一化

## TMDB 缓存

TMDB 响应缓存到 SQLite：

```text
search results: 7 days
details: 30 days
image configuration: 30 days
```

同一个 task 内重复查询要内存去重。

TV detail 使用 `seasons[].season_number` 和 `episode_count`，V1 不请求每集 detail。

## 命名语言

`namingLanguage` 是全局业务设置，影响：

- TMDB 请求语言
- 最终命名标题
- `.nfo` 字段语言

UI language 与 namingLanguage 分离。

默认 namingLanguage 建议 `zh-CN`。

命名标题优先级：

1. TMDB localized title/name
2. fallback original_title/original_name

识别时仍比较 localized、original、alternative titles。

## 命名模板

V1 全局一套命名模板，不支持 library path override。

支持变量：

```text
{title}
{year}
{season}
{episode}
{season:00}
{episode:00}
```

默认模板：

```text
Movie folder:       {title} ({year})
Movie file:         {title}.{year}
TV root folder:     {title} ({year})
Season folder:      Season {season:00}
TV episode file:    {title}.{year}.s{season:00}e{episode:00}
```

保存 settings 时校验模板变量。

生成 plan 时再次校验模板。

渲染后的每个 path segment 必须 sanitize：

- 删除或替换非法路径字符
- 删除控制字符
- trim 首尾空白
- 删除 trailing dots
- title 内容不能生成 path separator

用户不能直接编辑 target path。

## Rename Plan 与执行

所有整理执行统一为：

```text
execute_rename_plan
```

自动和手动的区别由 `rename_plan.mode` 表达：

```text
auto | manual
```

### Plan Draft

手动处理以持久化 plan draft 为核心。

```text
rename_plan_drafts
- id
- libraryPathId
- libraryItemId nullable
- mode
- status: draft | submitted | expired
- rowsJson
- createdAt
- updatedAt
- expiresAt
```

手动流程：

1. 用户选择 item 或 item 内文件。
2. 后端实时读取 WebDAV 文件列表。
3. 后端尝试识别并生成 plan draft。
4. 前端展示 editable plan rows。
5. 用户确认或手动指定 movie/TV identity。
6. 用户修正 season/episode/movie mapping。
7. 后端 validate draft。
8. 用户提交。
9. 生成正式 rename plan。
10. 创建 `execute_rename_plan` task。

manual draft rows 可勾选/取消。

- unchecked row 不进入正式 plan。
- checked row 缺少必要信息时阻止提交。
- TV row 可修改 season/episode。
- target path 由 identity + mapping + template 计算，不可直接编辑。

### Rename Plan

正式 plan 可覆盖多个 library items。

```text
rename_plans
- id
- libraryPathId
- mode: auto | manual
- status
- templateSnapshotJson
- createdBy
- confirmedAt
- createdAt
```

```text
rename_plan_items
- id
- planId
- libraryItemId
- sourceFilePath
- targetFilePath
- targetTopLevelPath
- mediaKind: movie | tv
- sourceMediaId
- season
- episode
- overwrite
- status
```

手动详情页默认生成单 item plan。

批量“整理已识别”可以生成多 item plan。

执行前 worker 必须二次校验：

- 源视频文件仍存在。
- 目标目录可创建或已存在。
- 目标文件不存在，除非手动 force overwrite。
- library item 仍存在且 identity 未变。
- 使用 plan 内保存的 naming template snapshot，不受后续 settings 修改影响。

### 文件夹冲突与文件冲突

V1 不把目标 folder 已存在视为冲突。

只把目标文件路径已存在视为冲突。

Movie：

- 目标 folder 不存在：创建 folder。
- 目标 folder 已存在且目标视频文件不存在：移动进去。
- 目标视频文件存在：冲突。

TV：

- 目标 show folder/season folder 不存在：创建。
- folder 已存在：复用。
- 目标 episode 文件存在：冲突。

自动整理永不 overwrite。

force overwrite 只允许手动确认流程。

### 执行策略

批量 plan 执行尽量继续：

- 一个文件失败不阻塞其它文件。
- 一个 item 失败不阻塞其它 items。

task 结果：

```text
全成功 -> succeeded
部分成功部分失败 -> partially_failed
全失败 -> failed
```

文件级 execution records 必须记录：

- source path
- target path
- status
- error
- overwritten target path
- sidecar results
- metadata results
- timestamps

一级 file item 整理后：

- 旧 file item 本地记录硬删除。
- 目标 folder item 新建或 upsert。
- 两者不是同一个实体，不做 file -> folder 身份转换。

organized item 内部部分整理失败：

- item `status` 保持 `organized`
- 用 execution summary/health 表达失败
- 只有读取/计划级整体失败才可设 `failed`

## Sidecar Files

与视频同 basename 的 sidecar 跟随视频移动：

```text
.srt .ass .ssa .vtt .sub .nfo
```

规则：

- plan draft 显示 sidecar preview，可折叠，不可取消。
- rename plan 保存 draft 生成时的 sidecar snapshot。
- worker 执行时重新发现实际存在的 sidecar。
- sidecar 继承视频 overwrite 策略。
- sidecar 移动失败不阻止视频被视为整理成功。
- sidecar 失败写 logs 和 execution detail。

V1 不移动源目录内已有的 `poster.jpg` 或目录级 artwork。

## Metadata

全局设置：

```text
metadataEnabled
overwriteMetadata
```

metadata 只在 rename 成功后生成。

metadata 失败不回滚 MOVE，不改变 organized 状态。

生成内容：

```text
Movie:
  movie nfo using movie filename basename
  poster.jpg

TV:
  tvshow.nfo
  poster.jpg
```

V1 不生成：

- episode `.nfo`
- season poster
- backdrop
- fanart
- banners
- downloaded subtitles

metadata 写入按目标顶级 folder 去重，不按文件重复执行。

默认不覆盖已有 metadata。`overwriteMetadata=true` 时可覆盖。

## Cleanup Invalid Directories

每个 library path 提供手动“清理无效目录”。

它实时扫描远端目录并生成删除计划，不依赖 `library_items` 状态。

候选目录：

- 目录及后代不含有效视频文件。
- 目录及后代不含有效字幕文件。
- library root 本身永不删除。

有效字幕扩展：

```text
.srt .ass .ssa .vtt .sub
```

preview 显示：

- candidate directory
- file count
- subdir count

确认后创建 `cleanup_invalid_dirs` task，逐项删除并写 logs/execution records。

## 任务系统

V1 使用 SQLite-backed persistent task queue。

web 创建 task，worker 执行 task。

task types：

```text
scan_library_path
scan_library_item
execute_rename_plan
cleanup_invalid_dirs
```

task states：

```text
queued
running
succeeded
partially_failed
failed
```

执行规则：

- 全局同一时间只运行一个后台任务。
- FIFO。
- V1 不支持任务取消。
- 外部服务失败不做复杂自动 retry。
- 小规模 HTTP transient retry 允许。
- 429、403 等应记录并 surfaced 到 task status。

worker 启动时：

- 遗留 running task 标记为 failed。
- 写英文 log。
- 不做精细恢复。

### Scheduler

Scheduler 与 Executor 分开建模。

- Scheduler 只负责检查 library path 是否到点，并创建 `scan_library_path` task。
- Executor 只负责 claim queued task 并执行 handler。
- Scheduler 不直接扫描 WebDAV。

定时扫描默认关闭。

建议选项：

```text
off
every 6 hours
every 12 hours
daily
```

如果 scheduled scan 触发时同一 library 已有 queued/running scan，跳过并写 log。

### web/worker 通信

V1 只通过 SQLite 通信。

- web insert task
- worker claim task
- worker update progress/result
- worker insert logs
- web 读取 task/log/library 状态

不加 HTTP worker API、IPC、Redis、pub/sub。

worker 可 1s 轮询 queue。

## 手动处理与待确认

所有 `pending_review` UI 都提供手动搜索 TMDB 并指定 identity 的流程，不管是否有系统候选。

用户可：

- 使用系统候选
- 手动搜索并指定其它 TMDB 结果
- 重新识别

手动指定 identity 后：

- 可只保存为 `identified`
- 不强制马上整理
- 也可继续生成 plan draft 并执行

`identified` / `organized` 才开放手动整理入口。

`failed` 通过重新生成 plan draft 或手动指定 identity 处理，不重放旧 task。

## 批量操作

Library detail 顶部“整理已识别”只处理当前 library path 下的 `identified` items。

不包含：

- organized non-compliant
- unidentified
- pending_review
- failed

organized non-compliant 不提供单独批量入口。用户可在 item detail 手动整理，也可由自动整理处理。

## 自动整理

Library path 可开启 automatic organization。

扫描时：

- unidentified high-confidence item 可进入 identified 并生成 auto rename plan。
- organized item 跳过 TMDB 查询，但可检查 non-compliant 文件。
- organized item 中可明确解析的 non-compliant 文件，在 auto organize enabled 时可自动生成 plan。
- 不可明确解析的文件跳过，保留为 non-compliant。

自动整理：

- 创建 `rename_plan(mode='auto')`
- confirmedAt 自动设置
- enqueue `execute_rename_plan`
- 永不 overwrite
