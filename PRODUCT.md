# Renarr Product Context

## 产品定位

Renarr 是一个自托管的深色媒体库管理台，面向使用 WebDAV、AList、NAS 或类似远程文件系统维护影视库的用户。它的核心价值不是播放媒体，而是让用户可靠地完成媒体库整理工作：配置数据源、扫描远程目录、识别电影和剧集、匹配 TMDB 身份、生成重命名计划、执行整理任务，并持续观察任务与日志状态。

Renarr 应被设计成一个可长期打开、反复使用的操作工具。界面应让用户快速判断库状态、发现待处理项目、进入具体文件夹核对真实远端文件，并在确认后执行有风险的整理动作。

## 主要用户

- 自托管影视库维护者：维护电影、剧集目录，关心命名规范、目录结构和远程文件状态。
- NAS / WebDAV / AList 用户：需要连接远程存储，并以远端目录事实为准进行扫描与整理。
- 媒体库管理员：希望批量处理待识别、整理失败、命名不合规和任务异常项目。

## 核心任务

- 配置 WebDAV 数据源、媒体库路径和 TMDB 设置。
- 浏览媒体库的顶级目录与文件摘要，识别 pending、failed、organized 等状态。
- 对单个 item 实时查看远端文件，手动指定 TMDB 身份或重新扫描。
- 生成、检查和调整重命名计划，再提交后台任务执行。
- 通过系统任务与日志追踪扫描、整理和清理流程。

## 产品气质

Renarr 的气质是影视资料馆式的管理工具：有明确的媒体语境和资料整理感，但所有视觉表达都服务于管理效率。它可以使用海报、片名、年份、季集信息和媒体状态来建立影视库感知，但不应变成流媒体消费首页。

设计关键词：

- 深色
- 克制
- 稳定
- 有媒体资料感
- 面向重复操作
- 结果可核对

## 反例

Renarr 不应像以下类型：

- 以播放和推荐为中心的流媒体娱乐应用。
- 以营销介绍、巨幅 hero 和装饰卡片为主的网站。
- 只追求炫技动效、强烈渐变或沉浸氛围的视觉实验。
- 把添加表单、配置入口和危险操作长期堆在内容卡片里的后台页面。
- 只在前端临时切换状态、没有真实后端持久化的假控制台。

## 信息架构原则

- 左侧导航承载媒体库路径、系统任务、日志和设置等长期入口。
- 同一管理对象只保留一个主要维护位置；旧 URL 如需保留，应重定向。
- “添加”类操作放在对话框中，避免常驻内容区。
- “修改”类操作优先在原位置内联完成，并提供明确保存动作。
- 二值设置使用可操作开关，并接入真实 API 持久化。
- 列表摘要来自扫描缓存；详情页涉及文件时应实时读取 WebDAV。

## 视觉与交互方向

- 默认深色管理台风格，不提供主题切换入口。
- 使用 shadcn-svelte 作为基础组件体系；局部视觉差异优先通过页面级 class、主题变量或项目组件封装完成。
- 保持内容密度适中，优先支持扫描、对比、筛选和批量判断。
- 状态色应克制，避免把整个界面染成单一色系。
- 海报和媒体信息用于帮助识别对象，不应压过任务状态、文件统计和操作确认。
- 危险或不可逆操作必须让用户能看清影响范围。

## 可访问性基线

Renarr 当前不额外制定高于组件库默认能力的无障碍强化目标。实现时仍应保持基础语义、键盘可达、可见焦点、可读对比度和明确的错误反馈，不应为了视觉效果破坏 shadcn-svelte 组件自带的可访问性行为。

## 语言与语气

- 普通 UI 面向中文用户，使用简洁、明确的中文。
- 后端、worker 和日志消息保留英文原文。
- UI 错误通过错误码映射为中文，避免直接暴露技术堆栈。
- 文案应直接说明状态、影响和下一步，不使用营销口吻。
